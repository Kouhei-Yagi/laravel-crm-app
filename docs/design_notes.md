# 設計メモ（CRM システム）

本ドキュメントは、設計中に検討した内容・判断理由・迷った点を記録するための内部資料です。
後から設計意図を理解しやすくすることを目的としています。

---

## 1. customers テーブルについて

### ● 住所・電話番号を保持する理由

- 中小企業の CRM では、訪問営業・郵送物の送付などで住所・電話番号が必要になるケースが多い
- 住所は都道府県・市区町村・番地を分けるほどの要件はないため、単一カラムで管理する
- 郵便番号は`string(7)`とし、`digits:7`でバリデーションする（先頭 0 の問題を避けるため）

### ● ランク（rank）を A/B/C にした理由

- シンプルで運用しやすい
- 中小企業の営業現場でよく使われる分類方法
- `master_values.md`に定義済み

### ● assigned_user_id を必須にした理由

- 顧客の「初期対応者」を明確にするため
- データの責任範囲を明確にし、担当者不明のデータを防ぐため
- 実務では担当者が不明な顧客は運用上のリスクが高い
- 担当者削除によるデータ消失を防ぐため、外部キーは`restrictOnDelete()`を採用

---

## 2. projects テーブルについて

### ● project_status を持たせた理由

- 案件の進捗管理が必要なため
- 「見積中 → 提案中 → 契約済 → 失注 → 保留」という一般的な営業フローを想定

### ● customer_id を必須にした理由

- 案件は必ず顧客に紐づくため
- 外部キーは`restrictOnDelete()`を採用し、顧客削除で案件が消えることを防ぐ

### ● assigned_user_id を必須にした理由

- 案件の担当者を明確にするため
- 担当者削除で案件が消えるのは危険なため、`restrictOnDelete()`を採用

---

## 3. interactions テーブルについて

### ● project_id を nullable にした理由

- 案件に紐づかない問い合わせ対応（電話・メール）が存在するため
- 案件削除後も履歴を残すため、外部キーは`nullOnDelete()`を採用

### ● customer_id を必須にした理由

- 対応履歴は必ず顧客に紐づくため
- 顧客削除で履歴が消えるのは危険なため、`restrictOnDelete()`を採用

### ● assigned_user_id を必須にした理由

- 「誰が対応したか」は履歴として必須の情報
- 担当者削除で履歴が消えるのは危険なため、`restrictOnDelete()`を採用

### ● type（対応種別）を master 管理にした理由

- UI のプルダウンで統一し、入力ミスを防ぐため。

---

## 4. users テーブルについて

### ● Breeze 標準構成を採用した理由

- 認証機能を最小コストで実装できるため
- 標準的な`users`テーブル構造で十分

### ● ユーザー削除の扱い

- 担当者削除は`restrictOnDelete()`により禁止される
- 実務では退職者は「削除」ではなく 無効化（`active`フラグ）で対応するのが一般的
- 必要になれば`active`カラムの追加を検討

## 5. 外部キー制約の方針

本システムでは、実務的なデータ整合性を重視し、外部キー制約を以下の方針で統一しています。

- 顧客・案件・対応履歴の担当者はすべて restrictOnDelete()
- 顧客削除で案件や履歴が消えるのを防ぐため restrictOnDelete()
- 案件削除後も履歴を残すため interactions.project_id のみ nullOnDelete()

| テーブル     | カラム           | 制約         | 理由                             |
| ------------ | ---------------- | ------------ | -------------------------------- |
| customers    | assigned_user_id | restrict     | 担当者削除で顧客が消えるのを防ぐ |
| projects     | customer_id      | restrict     | 顧客削除で案件が消えるのは危険   |
| projects     | assigned_user_id | restrict     | 担当者削除で案件が消えるのは危険 |
| interactions | customer_id      | restrict     | 顧客削除で履歴が消えるのは危険   |
| interactions | project_id       | nullOnDelete | 案件削除後も履歴を残すため       |
| interactions | assigned_user_id | restrict     | 担当者削除で履歴が消えるのは危険 |
